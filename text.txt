## 1. What trade-offs did you consciously make due to time constraints?

Due to time constraints, I focused on delivering a stable and functional core experience using Next.js, TypeScript, Redux Toolkit (RTK Query), and Barikoi APIs, while postponing some advanced optimizations.

The key trade-offs I made include:

- **Used RTK Query with basic configuration**  
  I implemented API calls using Redux Toolkit Query, which provides built-in caching and state management. However, I did not fully optimize it with features like cache invalidation strategies, polling, or advanced query lifecycle handling.

- **Focused on core features over extended functionality**  
  I prioritized essential features such as:
  - location search
  - map rendering using react-bkoi-gl
  - selecting locations from the map
  - route direction between two points  
  Additional features like history, or saved locations were not implemented.

- **Simplified UI/UX design**  
  I used Tailwind CSS and shadcn components to build a clean interface quickly, without adding advanced animations or design systems.

- **Limited optimization of API calls**  
  While debouncing is used in search input, I did not implement request cancellation, rate limiting, or fine-grained caching control.

- **Basic error handling**  
  Only essential validation and fallback handling are included. A more robust system would include user-friendly error states and retry mechanisms.

These trade-offs allowed me to complete the core requirements efficiently while keeping the codebase clean and maintainable.


## 2. Refactoring for scalability

If this application needed to scale (more users, more data, additional features), I would focus on the following improvements:

- **API layer abstraction**
  Currently, API calls are directly used in components or simple utilities. I would introduce a dedicated service layer (or hooks) to centralize API logic and improve maintainability.

- **Introduce caching and data fetching optimization**
  I would integrate tools like React Query or SWR to:
  - cache API responses
  - reduce redundant requests
  - improve performance and UX

- **Improve state management structure**
  As the app grows, I would refactor Redux into a more modular architecture with clearly separated slices and normalized state.

- **Map performance optimization**
  For handling many markers or large datasets:
  - clustering markers
  - lazy loading map data
  - limiting re-renders

- **Component modularization**
  Break down large components into smaller reusable ones to improve readability and scalability.

- **Environment & API key security**
  Move all API handling to a backend proxy (Next.js API routes) to fully secure API keys and control usage.

- **Testing**
  Add unit and integration tests to ensure stability as new features are added.

These changes would make the application more maintainable, performant, and production-ready.
